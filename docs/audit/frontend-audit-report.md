# Frontend Audit Report — XKeen Control Panel

Дата: 2026-07-10
Скоуп: весь фронтенд `frontend/` (Svelte 5 / TypeScript, ~35 200 строк в 60+ компонентах): дизайн-система и стили, доступность, производительность, адаптивность, архитектура кода, UX/контент. Backend (Go) — вне скоупа, кроме мест, где контракты API влияют на фронтенд.

Метод: статический анализ кода, production-сборка (`vite build`), расчёт контрастов по токенам, рендер ключевых экранов в Chromium (vite preview + Playwright с замоканным API, скриншоты в `docs/audit/img/`), инвентаризация предупреждений `svelte-check`.

---

## Executive Summary

Панель функционально зрелая: 17 страниц, редактор конфигов с JSON-schema-валидацией, конструкторы Xray/Mihomo, PWA, i18n (ru/en), 24 e2e-спеки Playwright с моками API. Заложены правильные основы: CSS-токены, компоненты `Card/Button/Modal/Input`, обёртка `apiFetch` с CSRF, стор тостов/подтверждений. Тёмная тема на десктопе выглядит цельно.

Проблема в том, что почти каждая из этих основ **не доведена до применения**: `apiFetch` не используется ни разу (162 сырых `fetch`, CSRF-логика скопирована в 18 файлов), `Input.svelte` не используется ни разу (152 сырых `<input>`, 61 несвязанный `label`), `<Button>` использован 11 раз против 177 сырых `.btn`, а фича сворачивания сайдбара существует в виде пропсов и хоткея Ctrl+B, но не имеет ни кнопки, ни CSS. Параллельно жива копипаста: 5 файлов-гигантов по 2 000–3 500 строк, 7 копий `formatBytes`, задублированные функции между двумя конструкторами.

Два дефекта — реальные блокеры. **Мобильная вёрстка сломана на всех экранах ≤768px**: `.mobile-header` остаётся flex-элементом строки `.dashboard-layout` и превращается в тёмную колонку высотой во всю страницу, отжимая контент до 216px из 390px (панель позиционируется как PWA для телефона). **Светлая тема нечитаема в критичных местах**: тексты алертов имеют контраст 1.29–1.71:1, статусы «Запущено/Остановлено» — 1.85–1.95:1, потому что семантические цвета не переопределены для light. Обе проблемы не ловятся тестами: e2e гоняются только на Desktop Chrome и только в тёмной теме.

Рекомендованный фокус milestone: сначала два критических дефекта (мобильный каркас, палитра светлой темы), затем главная боль пользователей — редактор во весь экран (сейчас жёстко 520px высоты внутри контейнера 1320px — на 2560×1300 работает ~15% площади экрана), сайдбар/ИА меню, и производительность первой загрузки (монолитный бандл 1.63 MB / 490 KB gzip при заявленных в README «~160 KB»). Распил компонентов-гигантов сознательно вынесен за пределы milestone — это отдельная многосессионная работа.

## Ограничения аудита

- **Core Web Vitals (LCP/CLS/INP) не измерены** — нет доступа к прод-развёртыванию на реальном роутере и реальной сети. Все выводы по производительности основаны на фактическом размере сборки, составе чанков и коде (интервалы опроса, отсутствие ленивой загрузки). Полевые метрики **требуют замера** на устройстве (Lighthouse/WebPageTest через LAN).
- **Юнит-тесты (`vitest`) не прогнаны**: запуск в песочнице аудита упал на создании dev-сервера Vite (ограничение среды, не дефект репозитория). E2e-сьют Playwright целиком тоже не запускался; браузер использовался только для скриншот-сценариев с замоканным API.
- **Скриншоты сняты на `vite preview` с синтетическими данными** (моки API по образцу `tests/basic.spec.ts`); реальные данные могут дать другие крайние случаи (длинные имена узлов и т.п.).
- **Кроссбраузерность вживую не проверялась** (только Chromium; Safari/Firefox — по коду, без рендера).
- Расчёты контраста выполнены по значениям токенов из `global.css` (математика WCAG), а не по снятым с экрана пикселям; для градиентных фонов возможны отклонения ±.

## Findings

### [F-01] Мобильная вёрстка сломана: `.mobile-header` становится боковой колонкой
- **Направление:** Адаптивность и кроссбраузерность
- **Приоритет:** critical
- **Доказательство:** `frontend/src/styles/global.css:432` — `.dashboard-layout { display: flex }` (направление row); `global.css:657` — `.mobile-header { position: sticky }`. Sticky-элемент не выходит из потока, поэтому при ≤768px шапка становится **первым flex-элементом строки**. Замер в Chromium при вьюпорте 390×844: `.mobile-header` rect = `{x: 0, width: 173px, height: 2209px}` (колонка во всю высоту страницы), `.main-content`/`.container` = `{x: 173, width: 216px}`. Контент получает 216 из 390px. Скриншот: `docs/audit/img/dashboard-mobile-390.png`. E2e это не ловят: в `playwright.config.ts` единственный проект — Desktop Chrome.
- **Рекомендация:** перевести `.dashboard-layout` на `flex-direction: column` в мобильном брейкпоинте (или вынести шапку из flex-контейнера / `position: fixed` + отступ контенту). Добавить мобильный проект в Playwright (например, `devices['Pixel 7']`) со smoke-проверкой «ширина `.container` ≈ ширине вьюпорта».
- **Оценка усилий:** S

### [F-02] Светлая тема: контраст алертов и статусов 1.29–1.95:1 (WCAG 1.4.3 fail)
- **Направление:** Доступность
- **Приоритет:** critical
- **Доказательство:** цвета текста алертов захардкожены под тёмный фон и не переопределяются в `[data-theme='light']`: `.alert-success { color: #b6efce }` (`global.css:378-392`) → на белой карточке контраст **1.29:1**; `.alert-error #f4b6be` → **1.71:1**; `.alert-warning #f3d9a6` → **1.37:1**. Семантические токены `--success #46d18a`, `--warning #f0b450` тоже не переопределены для light (`global.css:98-125`): статусные тексты «Запущено/Остановлено» на Дашборде/Сервисах — **1.95:1 / 1.85:1** (норма AA — 4.5:1). Видно на `docs/audit/img/dashboard-light-1600.png` (бледные «Запущено», бадж «214 строк»). Т.е. именно сообщения об ошибках и статусы ядер — ключевая информация панели — нечитаемы.
- **Рекомендация:** ввести light-переопределения для `--success/--warning/--danger/--accent` (тёмные варианты: ~#1a7f4e, #9a6a00, #c22a3a) и перевести `.alert-*`/`.chip-*`/`.badge-*`/`.status-*` с хардкодов на токены. Добавить скрипт-проверку контраста пар «токен текста × токен фона» в CI (математика уже есть в этом аудите).
- **Оценка усилий:** M

### [F-03] Редактор конфигов: рабочая область жёстко 520px высоты внутри контейнера 1320px
- **Направление:** UX и контент
- **Приоритет:** high
- **Доказательство:** `frontend/src/Editor.svelte:1578` — `<div style="height: 520px; ...">` вокруг CodeMirror; страница обёрнута в `.container { max-width: 1320px; padding: 28px 36px 60px }` (`global.css:438-442`). На мониторе 2560×1300 редактор занимает ~960×520px — меньше 15% площади экрана, справа и снизу пустота (скриншот `docs/audit/img/editor-dark-2560.png`). Это прямое подтверждение жалобы «полезная область окна редактирования маленькая и неудобная». `CodeMirrorEditor.svelte:233` при этом задаёт `min-height: 500px` — конфликтующие источники высоты.
- **Рекомендация:** сделать editor-страницу full-bleed (без `max-width`) и колоночный flex-лейаут высотой `calc(100dvh - шапка)`: тулбары фиксированной высоты, CodeMirror — `flex: 1; min-height: 0`. Файловое дерево — сворачиваемая панель (уже есть кнопка). Проверить, что e2e `editor-*.spec.ts` не завязаны на фиксированную высоту.
- **Оценка усилий:** M

### [F-04] Подсветка синтаксиса CodeMirror — светлая палитра на почти чёрном фоне
- **Направление:** Доступность
- **Приоритет:** high
- **Доказательство:** `CodeMirrorEditor.svelte:158` — `syntaxHighlighting(defaultHighlightStyle, { fallback: true })`; `defaultHighlightStyle` из `@codemirror/language` рассчитан на светлый фон (ключевые слова `#708`, строки `#a11`, имена свойств `#219`), а фон редактора захардкожен `#050d16` (`CodeMirrorEditor.svelte:239`). Тёмно-синие имена свойств JSON на чёрном фоне практически невидимы — видно на `docs/audit/img/editor-dark-2560.png` (первая строка: часть токенов «проваливается» в фон, контраст порядка 1.5:1).
- **Рекомендация:** подключить тёмную тему подсветки (например, `@codemirror/theme-one-dark` или собственный `HighlightStyle` на токенах панели), переключаемую вместе с темой приложения; фоны редактора увязать с токенами, а не `#050d16 !important`.
- **Оценка усилий:** S

### [F-05] Фича «свернуть сайдбар» мертва: нет ни CSS, ни кнопки
- **Направление:** Архитектура и качество кода
- **Приоритет:** high
- **Доказательство:** `Dashboard.svelte:414-426` — состояние `isSidebarCollapsed`, запись в `localStorage`, хоткей Ctrl+B; `Dashboard.svelte:510` — `class:sb-collapsed`. Класс `.sb-collapsed` **не определён ни в одном CSS** (grep по проекту — 1 вхождение, само место навешивания). `Sidebar.svelte:14-15` принимает `isCollapsed`/`onToggleCollapse` и **нигде их не использует** — кнопки сворачивания в разметке нет. Итог: Ctrl+B молча пишет в localStorage и ничего не делает.
- **Рекомендация:** решить судьбу фичи: либо реализовать (иконочный rail ~64px: CSS для `.sb-collapsed`, кнопка в футере сайдбара, тултипы для пунктов), либо удалить весь мёртвый код. Реализация закрывает и часть жалобы про «неудобное меню».
- **Оценка усилий:** M

### [F-06] Информационная архитектура меню: «Настройки» дважды, мониторинг разорван по трём группам
- **Направление:** UX и контент
- **Приоритет:** high
- **Доказательство:** `Sidebar.svelte:105-114` — «Настройки» в группе «Основное»; `Sidebar.svelte:345-355` — те же «Настройки» второй раз в группе «Доступ» с захардкоженным тегом `превью` (не i18n). Мониторинговые разделы разбросаны: «Подключения» — в «Сервисах» (`:160`), «Прокси/Правила» — в «Прокси и правила» (`:219-253`), «Трафик/Лимиты» — в «Инструментах» (`:290-311`). Пять `<details open>` без сохранения состояния — меню всегда полностью развёрнуто (15 пунктов + 5 заголовков). Подтверждает жалобу «поля в меню не в том порядке».
- **Рекомендация:** один пункт «Настройки»; перегруппировать по задачам пользователя (например: Обзор / Прокси и подписки / Маршрутизация / Наблюдение: Подключения+Трафик+Логи / Система: Сервисы+DAT+Консоль+Сеть+Настройки); сохранять open/closed групп в localStorage; добавить `aria-current="page"` активному пункту (сейчас 0 вхождений `aria-current` в проекте).
- **Оценка усилий:** M

### [F-07] Двуязычие фактически сломано: ~500 захардкоженных русских строк при живом i18n
- **Направление:** UX и контент
- **Приоритет:** high
- **Доказательство:** локали полные (`locales/ru.json`/`en.json` по 1080 строк), переключатель языка есть (`Settings.svelte:517`), но в разметке ~530 кириллических строковых литералов: XrayRoutingConstructor — 80, MihomoGenerator — 77, Proxies — 65, Editor — 45 (примеры: `Editor.svelte:1511` «Скачать файл», `:1525` «Шаблоны», `:1641` «Схема», `:1652` «Эксперт», `:1689` «Ctrl+S — сохранить»; `Dashboard.svelte:1202` «хвост последних 500 строк», `:1226` «shell в окружении XKeen»). Плюс 68 паттернов `$t('key') || 'русский фолбэк'` (признак отсутствующих ключей) и aria-атрибуты на разных языках: `aria-label="Выполнено"` (`Dashboard.svelte:624`) против `aria-label="Close"` (`Modal.svelte:85`). `index.html:2` — `lang="ru"` навсегда, `document.documentElement.lang` при смене языка не обновляется (0 вхождений) — нарушение WCAG 3.1.1 для EN-пользователей.
- **Рекомендация:** вынести строки в локали по страницам (начать с топ-4 файлов = 267 строк), запретить кириллицу в `.svelte` ESLint-правилом (`no-restricted-syntax` по диапазону U+0400–U+04FF вне `t()`), обновлять `<html lang>` в `setLang`, локализовать aria-label. Тест `tests/i18n-missing-keys.cjs` расширить проверкой фолбэков `|| '...'`.
- **Оценка усилий:** L

### [F-08] Монолитный бандл 1.63 MB (490 KB gzip): код-сплиттинг отключён вручную
- **Направление:** Производительность
- **Приоритет:** high
- **Доказательство:** факт сборки: `dist/assets/index-*.js = 1 626.00 kB │ gzip: 489.58 kB` + CSS 170 KB (27 KB gzip). Причины: `vite.config.ts:12` — `manualChunks: undefined` (сплиттинг выключен осознанно); `Dashboard.svelte:19-33` — все 17 страниц импортированы статически, включая Editor→CodeMirror+`codemirror-json-schema` (тянет shiki-грамматику ~198 KB) и оба конструктора по 3.4К строк. README при этом заявляет «bundle ~160 KB gzipped» (`README.md:10`) — расхождение в 3 раза. Для панели, которую раздаёт роутер с 64–128 MB RAM по Wi-Fi, это прямой удар по первой загрузке; LCP/INP — требуют замера, но 1.6 MB парсинга JS на бюджетном телефоне — заведомо секунды.
- **Рекомендация:** убрать `manualChunks: undefined`; перевести страницы на динамические `import()` (svelte: `{#await import('./Editor.svelte')}`), в первую очередь Editor, оба конструктора, DATManager; цель — первый экран (login+dashboard+shell) ≤200 KB gzip. Обновить цифру в README по факту.
- **Оценка усилий:** M

### [F-09] Поллинг-шторм: до ~1 запроса/сек всегда, без паузы для скрытой вкладки
- **Направление:** Производительность
- **Приоритет:** high
- **Доказательство:** `Dashboard.svelte:491-494` — шелл держит 4 интервала независимо от активной вкладки: `fetchLiveStatus` каждые 10с (внутри — до 4 HTTP-запросов: `service/status`, `mihomo/status`, `connections`, `kernels`), `fetchSystemStats` каждые 5с, `fetchCapabilities` каждые 10с, `fetchSubscriptionSummary` каждые 30с. Открытая страница «Сервисы» добавляет свои 2с/5с/15с (`Services.svelte`). Обработчиков `visibilitychange`/`document.hidden` в проекте **ноль** (grep) — свёрнутая PWA/фоновая вкладка продолжает молотить роутер и жрать батарею. Данные при этом нужны только Дашборду и бейджу сайдбара.
- **Рекомендация:** единый poll-менеджер: подписка по требованию (какая вкладка видна — те данные и опрашиваются), пауза при `document.hidden`, немедленный рефреш при возврате видимости; для Дашборда объединить 3-4 эндпоинта в один агрегирующий запрос на бэке (вне скоупа фронта — опционально).
- **Оценка усилий:** M

### [F-10] Инфраструктура API-клиента не используется: 162 сырых fetch, CSRF скопирован в 18 файлов
- **Направление:** Архитектура и качество кода
- **Приоритет:** high
- **Доказательство:** `lib/api.ts` предоставляет `apiFetch`/`apiFetchJSON` (CSRF + разбор envelope) — **0 импортов** во всём проекте (grep). 162 вызова `fetch(` в компонентах; `localStorage.getItem('csrf_token')` продублирован в 18 файлах; разворачивание envelope сделано ad-hoc и по-разному: `Array.isArray(envelope) ? envelope : (envelope.data ?? [])` (`Dashboard.svelte:149`), `envelope.data ?? envelope` (`stores.ts:46`), сырой `res.text()` с try-parse (`Dashboard.svelte:205-220`). Ошибки то тостятся, то глотаются `catch (_) {}`.
- **Рекомендация:** механическая миграция всех вызовов на `apiFetch/apiFetchJSON` (страница за страницей), единая обработка 401 (разлогин) и сетевых ошибок; ESLint-запрет на голый `fetch(` вне `lib/api.ts`.
- **Оценка усилий:** L

### [F-11] Компоненты-гиганты и межфайловая копипаста
- **Направление:** Архитектура и качество кода
- **Приоритет:** high
- **Доказательство:** размеры: `XrayRoutingConstructor.svelte` 3513 строк, `MihomoGenerator.svelte` 3398, `Editor.svelte` 2862, `Settings.svelte` 2225, `Proxies.svelte` 2091 (+ ещё 6 файлов >1000). Между двумя конструкторами задублированы одноимённые функции: `checkUndo`, `handleUndo`, `enableDNSRedirect`, `loadSchema`, `applyPreset`, `openImportModal/closeImportModal/parseImportLink`, `getNodeServer`, `getNodePort`, `addRule/removeRule/moveRule`, `openInEditor` (см. инвентаризацию функций обоих файлов). `formatBytes` существует в 7 экземплярах с расходящимися реализациями: `lib/format.ts` (пороговая) vs `Dashboard.svelte:331` (логарифмическая) vs ещё 5 локальных копий.
- **Рекомендация:** (а) немедленно: свести дубли утилит в `lib/` (format, undo-стек, DNS-redirect, import-link-парсер) — S/M; (б) отдельным milestone: распил гигантов на подкомпоненты по образцу `components/editor/*` (уже начатый правильный паттерн).
- **Оценка усилий:** L (распил — за пределами текущего milestone)

### [F-12] Дизайн-система: три слоя токенов-алиасов и двойная реализация каждого примитива
- **Направление:** Визуальная согласованность и дизайн-система
- **Приоритет:** high
- **Доказательство:** три поколения переменных живут одновременно: `--text/--text-secondary` (легаси), `--fg-primary/--fg-secondary/...` (текущие), `--color-text-primary/--color-primary-500/...` (алиасы «дизайн-системы», `global.css:50-58`). Примитивы задвоены: стили `.btn` определены в `global.css:260-316` **и** скопированы в `Button.svelte:32-86` (уже разошлись: у компонента в transition есть `filter`, у глобального — нет); `.card` — в `global.css:199-205` и `Card.svelte:35-42` (комментарий в Card.svelte утверждает «All visual rules now live in global.css», после чего правила переопределяются локально). Использование: 177 сырых `class="btn ..."` против 11 `<Button>`; сырые `.card` — включая `App.svelte`. Любая правка стиля кнопки теперь требует синхронных изменений в двух местах.
- **Рекомендация:** объявить единственный источник правды (предлагаю: глобальные классы, компоненты — тонкие обёртки без собственных копий стилей), удалить легаси-алиасы `--text*`/`--color-*` через кодмод, зафиксировать соглашение в `frontend/README`/CLAUDE.md.
- **Оценка усилий:** M

### [F-13] Клавиатурный фокус невидим на всех кнопках (WCAG 2.4.7)

### [F-21] Service Worker: cache-first для index.html без версии кэша + кэширование API
- **Направление:** Производительность
- **Приоритет:** medium

- **Оценка усилий:** M

## Предлагаемая структура Milestone для GSD

### Milestone: «UI Foundation: Mobile, Themes & Editor UX»
**Цель milestone:** панель корректно работает на телефоне и в обеих темах (AA-контраст), редактор использует весь экран, первая загрузка ≤200 KB gzip, база (фокус/формы/API-клиент) приведена к единому стандарту.

**Requirements** (для .gsd/REQUIREMENTS.md):
- REQ-1: На вьюпортах ≤768px контент занимает всю ширину экрана; мобильная шапка — фиксированная полоса сверху; в Playwright есть мобильный проект со смоук-проверкой геометрии. *(закрывает F-01, частично F-24)*
- REQ-2: Все текстовые пары «токен × фон» обеих тем проходят WCAG AA (4.5:1; крупный текст 3:1); алерты/статусы/чипы используют токены без сырых hex/rgba; проверка контраста токенов — скрипт в CI. *(F-02, F-16, F-19)*

- REQ-4: Подсветка синтаксиса в редакторе — тёмная тема с AA-контрастом токенов, синхронизирована с темой приложения. *(F-04)*
- REQ-5: Каждый интерактивный элемент имеет видимое `:focus-visible`-состояние; интерактивных `div` нет (заменены на button/a); Space и Enter работают одинаково. *(F-13, F-15)*

- REQ-10: 100% сетевых вызовов идут через `apiFetch/apiFetchJSON`; голый `fetch(` вне `lib/api.ts` запрещён линтером; 401 обрабатывается единообразно (разлогин). *(F-10)*
- REQ-11: Нативный `confirm()` не используется; `ConfirmDialog` поддерживает danger-вариант; деструктивные действия называют объект и последствие. *(F-22)*
- REQ-12: В `.svelte`-файлах нет захардкоженной кириллицы (ESLint-правило); `<html lang>` обновляется при смене языка; aria-label локализованы. Допустимо поэтапно: сначала топ-5 файлов. *(F-07)*

**Предлагаемые фазы (phases) roadmap** (каждая — один цикл discuss → plan → execute → verify → ship):
